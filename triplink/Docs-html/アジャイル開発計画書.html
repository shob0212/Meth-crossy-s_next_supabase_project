1. チーム体制と役割定義 (Roles)

アジャイル開発（スクラム）に基づき、役割を定義する。

役割 担当者 主な責任
プロダクトオーナー (PO) (名前) 「何を作るか」を決定する。機能の優先順位付け、仕様の最終決定者。
スクラムマスター (SM) (名前) 「チームの進行」を支援する。会議の進行、進捗確認、障害の除去。
開発メンバー (Dev) (全員) 設計、実装、テスト、デプロイを行う。

2. 全体ロードマップ (Roadmap)

全4フェーズ（Sprint 0〜4）でMVP（実用最小限の製品）の完成を目指す。

Sprint 0: 準備期間（Setup）

ゴール: 開発環境が整い、Hello Worldが表示される

  • 技術選定（言語・FW・DB・ホスティング）
  • GitHubリポジトリ作成・権限設定
  • 開発環境構築（Linter/Formatter含む）
  • データベース設計（ER図作成）
  • 画面遷移図（手書き等のラフでOK）の作成

Sprint 1: MVP開発（Core Value）

ゴール: ログインして、旅行を作成し、旅程を表示できる

  • ユーザー登録・ログイン機能の実装
  • 旅行プロジェクト作成機能（タイトル・日程）
  • タイムライン表示画面（ガワの作成）
  • 旅程データの追加・削除・編集機能

Sprint 2: 共有とUX向上（Sharing）

ゴール: 友だちを招待して、一緒に編集できる

  • 招待リンク生成機能
  • 招待されたユーザーの参加処理
  • 日付・時間入力UIの改善（UX向上）
  • スマホ・レスポンシブ対応の調整

Sprint 3: 支出管理（Wallet）

ゴール: 誰がいくら払ったか記録できる

  • 支出データの登録（支払者・金額・カテゴリ）
  • 支出リストの表示
  • 簡易的な合計金額の表示
  • （余裕があれば）簡単な割り勘計算ロジック

Sprint 4: 仕上げ・リリース（Release）

ゴール: 実際の旅行で使える品質にする

  • バグ修正・動作テスト
  • UIデザインの微調整（崩れの修正）
  • 本番環境へのデプロイ
  • ドキュメント整理・リリース

3. 開発サイクル（Sprint Routine）

1週間単位（1 Sprint）でサイクルを回す。

週間スケジュール例

曜日 イベント 内容
月 スプリント計画 今週の「ゴール」を決め、タスクをメンバーに割り振る。
火〜木 デイリースタンドアップ 毎日定時に15分実施。
1. 昨日やったこと
2. 今日やること
3. 困っていること
金 レビュー & レトロ レビュー: 動作するアプリを動かしてデモを行う。
レトロ (KPT): Keep/Problem/Tryで振り返りを行う。

4. タスク管理ルール (Workflow)

GitHub Projects / Trello / Notion 等を使用する。

ステータス定義

  1. Backlog: 将来やりたいこと全て。
  2. To Do: 今週のスプリントでやると決めたタスク。
  3. Doing: 現在作業中のタスク（原則1人1つまで）。
  4. Done: 実装・テストが完了し、メインブランチにマージされた状態。

タスク（チケット）の粒度

「機能単位」ではなく「ユーザー体験単位」で分割する。

  • ❌ 悪い例: 「旅程管理機能を作る」
  • ✅ 良い例: 「旅程の追加モーダルを表示する」「入力内容をDBに保存する」

5. 開発の掟 (Rules)

① Done（完了）の定義

「自分のPCで動く」は完了ではない。

  • GitHubのmainブランチにマージされていること。
  • デプロイ先の環境（または共有の開発環境）で動作確認できていること。

② MVP思考（完璧を目指さない）

  • 初期段階ではデザインの美しさより「動くこと」を優先する。
  • エラー処理や入力バリデーションは、最初は最低限で良い。

③ コミュニケーションのログ化

  • 仕様の相談や決定事項は、Slack/DiscordやIssueに残すこと（「言った言わない」を防ぐ）。
